# GumaPhoto 프로젝트 마스터 플랜 (가족 특화 시맨틱 검색 시스템)

이 문서는 GumaPhoto 서비스의 전체 기획과 향후 로드맵을 정의합니다. (수시로 업데이트 됩니다)

## 📌 최종 목표
- 가족 구성원의 얼굴을 학습시키고 모든 사진에서 인물, 표정, 배경(장소, 분위기) 정보를 다각도로 추출하여 벡터 DB에 영구 보존.
- 모바일(스마트폰)에서 무제한 대량의 사진을 즉각적으로 업로드하고, 백그라운드에서 실시간으로 갤러리를 정렬/분석하는 무중단 동기화 서비스 구축.
- 사용자의 자연어(예: "가족들이랑 해변가에서 웃고있는 사진", "아빠가 군복 입은 사진")를 찰떡같이 알아듣는 LLM 기반 사진 검색 엔진 개발.
- 홈서버(FastAPI+Nginx) 위에서 가족 누구나 웹/모바일로 쉽게 접근 가능한 구글 포토(클라우드 갤러리) 수준의 개인화 갤러리 런칭.

---

## ✅ 1단계: 개발 환경 구성 및 홈서버 인프라 구축 (완료)
1. **Docker 기반 환경 세팅**: `docker-compose.yml`을 통해 메인 FastAPI 애플리케이션(`gumaphoto_app`) 및 Qdrant 벡터 DB(`gumaphoto_qdrant`) 구축.
2. **홈서버 연동 지원**: `TheGumaLab` 메인 포털 인덱스에 GumaPhoto 추가 및 Nginx 리버스 프록시(`/GumaPhoto`) 매핑 완료.

## ✅ 2단계: 자동 스마트 분류 파이프라인(Organizer) (완료)
1. **시계열 & 메타데이터 구조화**: 기존 뒤섞인 원본 사진들을 `YYYY/YYYY-MM/YYYY-MM-DD_01.jpg` 형식의 규격화된 트리 구조로 자동 재배치. (`organizer_pipeline.py`)
2. **Junk(찌꺼기) 자동 필터링**: 쓸모 없는 스크린샷, 심하게 흔들린 사진, 완전 중복 사진 강제 삭제 및 용량 최적화 수행.
3. **폴더 컨텍스트 영구 보존**: 과거에 의미 있게 묶어둔 폴더명(예: `2005_성욱_군대`)을 DB에 `original_context`라는 이름으로 보존하여 추후 벡터 정보의 그라운드-트루스(Ground Truth)로 활용.

## ✅ 3단계: 멀티모달(얼굴+배경+사물+서사) 하이브리드 벡터화엔진 (완료)
1. **안면 인식 (InsightFace)**: 최대 수준의 안면 추출 엔진(`buffalo_l`)을 가동하여 사진에 등장한 인물들의 512차원 벡터와 성별/나이 추출.
2. **표정 분석 (HSEmotion)**: 안면 영역만 따로초경량 PyTorch 모델로 분석하여 'Happiness', 'Sadness', 'Contempt' 등 다채로운 감정 상태 분류. (과거 DeepFace 대체)
3. **분위기 & 배경 분석 (최신 SigLIP)**: `google/siglip-base-patch16-224` 모델을 가동하여 사진의 전반적인 분위기와 피사체 정보를 최첨단 다국어 기반 벡터(768차원)로 합축. (기존 CLIP 한계 돌파)
4. **정밀 객체 탐지 및 캡션 묘사 (Florence-2-base)**: 단순히 사물(car, dog)만 뽑던 YOLO를 퇴역시키고, 마이크로소프트의 VLM을 토대로 이미지의 구체적인 상황 묘사(Caption) 문장과 핵심 사물 태그(OD)를 동시에 추출.
5. **Qdrant 멀티-인덱스 하이브리드 아키텍처**: "scene(배경)", "face(얼굴)" 벡터 분리 및 JSON 형태의 캡션/위치/감정 메타데이터 페이로드를 조립하여 DB에 실시간 꽂아 넣음.

## 🚀 4단계: FastAPI 백엔드 검색 엔진 및 UI/UX 고도화 (예정)
1. **LLM 기반 하이브리드 마법 검색 엔진**: 사용자가 자연어로 입력한 검색어를 Gemini가 `people(인물)`, `location(보유 장소 목록과 대조한 정확한 지명)`, `objects(명확한 영어 사물)`, `scene(순수 영어 캡션)`으로 똑똑하게 파싱함. 이후 Qdrant의 Hard Filter(장소/객체 문자열) + SigLIP Vector Search(상황 공간벡터 70% 가중치) + BM25 Caption Overlap Score(캡션 단어 일치율 30% 가중치)를 합산하여 리랭킹(Re-ranking) 하는 극강의 하이브리드 아키텍처 달성.
2. **구글 포토(Google Photos) 스타일 타임라인 갤러리 뷰**: 검색을 하지 않고 홈 화면에 처음 들어왔을 때, 전체 사진을 연도별/월별로 매우 부드럽게 스크롤하며 감상할 수 있는 기본 타임라인 UI 설계.
3. **편리한 선택 및 앨범 저장 UI**: 검색된 사진 결과물 중 마음에 드는 사진들을 다중 선택(체크박스)하여 '내 스마트폰으로 다운로드' 하거나 '임시 앨범'에 묶어 저장할 수 있는 사용자 친화적 편의 기능 추가.

## 🚀 5단계: 모바일 실시간 업로드 및 동기화 무중단 파이프라인 (진행 중)
1. **사진 업로드 수신**: 스마트폰/웹에서 다중 파일 업로드 시 `data/uploads_raw` 임시 서버 폴더에 저장.
2. **자동 폴더 분류 (Organizer)**: 업로드된 사진의 Exif 메타데이터(촬일자/GPS)를 읽어 `YYYY-MM_Location` 같은 규격화된 네이밍 규칙(예: `2026-03_Seoul.jpg`)에 따라 새로운 폴더를 생성하거나 기존 폴더에 맞게 사진 자동 이동.
3. **실시간 AI 인덱싱 (AI Scan)**: 분류가 끝난 즉시 `vector_indexer.py`의 핵심 스캔 로직을 호출하여 `known_faces.pkl`을 바탕으로 한 인물(people)과 장소(location) 태그를 추출하고 LLM CLIP 벡터로 분석.
4. **Qdrant DB 즉각 반영 (Upsert)**: 메타데이터 분석이 끝나는 즉시 Qdrant DB에 벡터와 Payload 태그를 실시간 Point로 꽂아 넣어, 사용자가 사진을 업로드하자마자 즉시 검색 가능하게 함.

## 🤖 6단계: 사용자 참여형 AI 피드백 및 자가 학습 시스템 (Active Learning) (진행 중)
1. **사진 상세 모달 및 액션 버튼 추가**: 갤러리/검색 결과에서 사진 클릭 시 화면이 모달 창으로 확장되며 확대된 사진 표출. 하단에 `Download`, `Feedback`, `Delete` 3개의 기능 아이콘 배치.
2. **자연어 피드백 수집 (Text Box)**: 사용자가 `Feedback` 버튼을 누르면 피드백을 입력할 수 있는 텍스트 박스가 나타남. 사용자는 "이 사진은 송이가 아니고 성욱이야" 와 같이 자연어로 AI의 오류 지적 가능.
3. **학습 데이터(known_faces.pkl) 실시간 교정 및 메인 피사체 추출**: 사용자의 피드백을 LLM(Gemini)이 파싱하여 정정해야 할 인물(송이 -> 성욱)을 파악. 다중 인물이 있을 경우 **가장 넓은 면적을 차지하는 큰 얼굴(메인 피사체)**의 벡터를 추출하여 `known_faces.pkl`의 정답(성욱) 배열에 학습 데이터로 강제 편입.
4. **해당 사진 즉각 태그 갱신 (Immediate Local Update)**: 전체 사진 리스캔 작업은 무겁기 때문에 나중으로 미루되, **사용자가 피드백을 남긴 바로 그 사진**만큼은 즉시 Qdrant DB의 Payload 태그를 교체하여 검색 결과에 즉답하는 UI/UX 최적화 수행.
5. **전체 태그 리스캔(Rescan) 자동화 (Global Update)**: 학습 데이터가 변경됨에 따라 과거의 잘못 태깅된 모든 사진을 수정. 연산 비용이 크므로 **주 1회(Cron/Scheduler) 일괄 스캔**을 돌리거나, 관리자 옵션의 **'수동 재스캔 실행' 버튼**을 눌렀을 때만 작동되도록 백그라운드 워커 설계.

## 🛠️ 유지보수: 파일-DB 자가 치유 아키텍처 (예정)
1. **앱 내 완전 삭제 연동 (Hard Delete)**: 사용자가 프론트엔드 모달 창에서 `Delete(휴지통)` 버튼을 누르면, 백엔드 API가 호출되어 ① 서버 스토리지의 실제 이미지 파일(`os.remove`) 파기, ② Qdrant Vector DB의 해당 Point 완전 삭제, ③ SQLite `vectorized_files` 상태 테이블에서 영구 제외 처리를 하나의 트랜잭션으로 즉각 실행.
2. **주기적 시스템 안정성 배치 (CronJob)**: 혹시 모를 서버 다운이나 백그라운드 작업 중단 등에 대비하여, 정기적으로 서버 내 실제 보유 파일 리스트와 Qdrant 내부/SQLite 데이터를 대조(Diff)하고 어긋난 유령 데이터를 자동 청소(Garbage Collection)하는 시스템 유지보수 로직.
